# Phase 4.2 P42-D003 test and evidence gates

## Status and purpose

**PLANNED.** P42-D003 is **OPEN** with a **RECOMMENDED** architecture class pending independent review and explicit project-owner approval. This document defines future evidence gates; no test was implemented or run in this task, and no result is **VERIFIED** by this document.

The gates refine the accepted [Phase 4.2 test and evidence plan](phase42-test-and-evidence-plan.md) for the proposed composition. They trace to the accepted [confinement requirements](phase42-confinement-requirements.md), [resource-budget framework](phase42-resource-budget.md), approved [P42-D002 threat model](phase42-d002-threat-model.md), and open [decision register](phase42-open-decisions-and-handoff.md).

Evidence must be produced from the exact claimed platform profile. It must distinguish enforcement from observation, prevention from detection, authoritative process identity from numeric PID logging, ordinary IP-network denial from local-IPC isolation, verified paths from exact executed-byte identity, and architecture recommendation from implemented mechanism.

## Gate posture

Before P42-D003 can be approved, an independent reviewer must confirm that the proposed composition maps every applicable requirement to an enforcement owner, a failure behavior, and a reproducible evidence family. Approval would accept an architecture decision only; it would not prove implementation.

Before any later implementation or platform-support claim can be accepted, every applicable family below must produce the mandatory evidence on every claimed profile, from a restored clean state, with independent reproduction. A missing control, probe, observation hook, cleanup proof, or mandatory assurance record prevents success acceptance. An unsupported profile must refuse launch. Exact policy and numeric values remain **OPEN** under the cited decisions and budget records.

## Evidence dimensions

| Evidence class | What it must establish | Minimum artifacts | What it cannot establish alone |
| --- | --- | --- | --- |
| Unit evidence | Deterministic parsing, policy selection, bounds, categories, and fail-closed decisions without relying on validator cooperation. | Inputs, expected decisions, actual decisions, schema versions, and negative cases. | Kernel enforcement, host cleanup, or platform support. |
| Integration evidence | Correct interaction among orchestrator, external supervisor or launcher, transports, staging, and audit handling. | Correlated attempt timeline, transport identities, policy manifest, terminal category, and cleanup record. | That the kernel denied a direct hostile syscall. |
| Host-level evidence | Host-observed process trees, files, mounts, namespaces or equivalents, identities, handles, resource state, diagnostics, and residue. | Before-and-after censuses, authoritative ownership identity, configuration capture, and residual scans. | Cross-platform equivalence or resistance to a trusted-host compromise. |
| Kernel-enforcement evidence | The selected kernel or service mechanism prevents or terminates prohibited behavior under direct native-code attempts. | Negative operation trace, enforcement event, authoritative policy identity, and host-observed terminal state. | Correct orchestration, complete evidence, or suitability on another kernel profile. |
| Platform-profile evidence | The exact OS image, kernel, runtime, packages, configuration, and exposed capabilities support the complete composition. | Immutable profile identity, capability-probe output, policy identities, limitations, and unsupported-profile behavior. | Support for a distribution family, another release, or another environment by analogy. |
| Independent reproduction | A reviewer can reconstruct and repeat the procedure without ambient checkout or host state. | Exact commit and artifact identities, clean-room recipe, restored-state proof, raw bounded evidence, and signed or otherwise attributable review record. | Absolute security, external certification, or production readiness beyond the reproduced profile. |

## Cross-family acceptance gates

The future evidence package is acceptable only if all of the following hold:

1. Every exact claimed platform profile has all applicable unit, integration, host-level, kernel-enforcement, and platform-profile evidence.
2. An independent reviewer reconstructs the exact commit, artifacts, host profile, kernel, runtime, packages, configuration, policies, and capability results from restored clean state.
3. Negative tests exercise hostile native code and descendants; validator-supplied logs are never the sole proof of enforcement.
4. Every terminal path includes process, workspace, IPC, transport, policy, dump, diagnostic, descriptor, mount, and residual-state verification.
5. Any missing required control or evidence item refuses launch or prevents success acceptance.
6. Numeric budgets are approved separately and every referenced value is bound to the exact profile and policy identity.
7. P42-D001 and the applicable mechanism decisions are resolved before a production-platform claim.
8. The evidence package contains no inference that a container, VM, namespace, seccomp filter, MAC policy, rootless identity, low-privilege account, or JavaScript control is independently sufficient.
